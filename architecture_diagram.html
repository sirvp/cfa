⚡ Utility Company Review Intelligence — System Architecture

End-to-end pipeline: App Store, Play Store & Trustpilot → AI Analysis → Live Dashboard

🕐 Built in ~6 Hours 🤖 Claude Haiku AI 📅 Runs Daily at 06:00 ☁️ Streamlit Cloud
⚙️

Fully Automated Daily Pipeline — Zero Manual Steps

Windows Task Scheduler fires run_daily.bat at 06:00 every morning → scrapes fresh reviews → AI-analyses them → exports to GitHub → Streamlit Cloud picks up the change and the dashboard is live with today's data.

Layer 1 — Data Sources
🌐 External Review Platforms

Apple App Store (iOS)

  • RSS feed: itunes.apple.com/gb/rss/customerreviews/id=562202559
  • Public, no authentication required
  • 10 pages × 50 reviews = 500 reviews max
  • Filtered to last 180 days — GB region

Google Play Store (Android)

  • Library: google-play-scraper (handles pagination tokens)
  • App ID: uk.co.scottishpower
  • Batch size: 200 reviews per page
  • Filtered to last 180 days — UK region

Trustpilot

  • Web scrape: trustpilot.com/review/www.scottishpower.co.uk
  • Paginated review pages; public, no auth required
  • Filtered to last 180 days
  • Captures: star rating, review body, date, author
Layer 2 — Scrapers
🕸️ Data Collection Scripts

AppStoreScraper_JSON.py

  • Paginates RSS (pages 1–10); skips page 1 entry 0 (metadata)
  • Stops early once oldest review pre-dates the 180-day window
  • Handles missing fields (rating-only, no text)
  • Output: Reviews/Utility Company_AppStore_Reviews_YYYY-MM-DD.json

PlayStoreScraper_JSON.py

  • Follows continuation tokens until data exhausted or cutoff hit
  • Normalises field names, fills missing content with empty strings
  • Converts all dates to ISO format before writing
  • Output: Reviews/Utility Company_Playstore_Reviews_YYYY-MM-DD.json

TrustpilotScraper_JSON.py

  • Paginates HTML review pages with BeautifulSoup
  • Stops early once reviews pre-date the 180-day window
  • Normalises star ratings (1–5) from Trustpilot score classes
  • Output: Reviews/Utility Company_Trustpilot_Reviews_YYYY-MM-DD.json

Layer 3 — Ingestion & Deduplication
📥 ingest.py — Smart Incremental Load into SQLite

1. SHA-256 ID

  • Stable review identity: hash(username + date + content)
  • Prevents duplicates across multiple runs

2. Field Normalisation

  • Maps AppStore & Play Store field names to shared schema
  • author, rating, body, app_version, date_posted

3. Cursor Filtering

  • Tracks last_fetched_date per source in DB
  • Only processes reviews newer than the cursor

4. Deduplication

  • UNIQUE constraint on (source, review_id)
  • Logs: inserted / skipped-old / skipped-dup
Layer 4 — AI Analysis
🤖 analyse.py — Claude Haiku (claude-haiku-4-5) via Anthropic API

Prompt Construction

  • Batches of 25 unprocessed reviews at a time
  • Sends review text + star rating
  • Provides fixed topic taxonomy to classify against:
  • login · billing · app_crash · smart_meter · customer_service · outage · account · other
  • Expects JSON back: { topics, sentiment, insight }

Response Handling

  • Direct JSON parse; fallback regex if Claude wraps in markdown
  • Validates sentiment ∈ { positive, neutral, negative }
  • Hallucination filter: topics stripped to allowed list only
  • Defaults to ["other"] if no valid topics remain
  • Crash-resilient: each review committed individually; failed ones retried next run
anthropic SDK claude-haiku-4-5 batch size: 25 max_tokens: 256
Layer 5 — Publish & Export
🚀 run_pipeline.py — Export CSV → Git Commit → Push to GitHub

CSV Export

  • SQL join: raw_reviews + insights (processed only)
  • Writes data/reviews_analysed.csv (committed to repo)
  • Database file (reviews.db) is gitignored — CSV is the publish artefact

Auto Git Push

  • Runs git add → commit → push origin main via subprocess
  • Commit message: chore: daily review data refresh YYYY-MM-DD
  • Streamlit Cloud detects the commit and redeploys automatically
Layer 6 — Live Dashboard
📊 dashboard.py — Interactive Streamlit Web Application

Data Loading

  • Prefers local reviews.db
  • Falls back to reviews_analysed.csv for Streamlit Cloud
  • 60-second cache TTL

Sidebar Filters

  • Store (App Store / Play Store)
  • Sentiment, Star rating
  • Topics (multi-select)
  • Date range picker

KPIs & Charts

  • Total, Positive%, Negative%, Avg ★
  • Sentiment over time (weekly stacked bar)
  • Rating distribution
  • Topics breakdown + Store split donut

AI Executive Summary

  • Calls Claude with period stats + prior-period comparison
  • Returns: snapshot, vs-prior, top issues, key change, recommendation
  • Cached; refreshable on demand

Data Layer — SQLite Database (reviews.db)
🗄️ Three Tables — Persistent Local Store
Table Key Columns Purpose
raw_reviews id, source, review_id (UNIQUE), author, rating, body, app_version, date_posted, is_processed Normalised reviews from both stores; is_processed=0 flags work for the AI analyser
insights review_id (FK), source, topics (JSON array), sentiment, insight, processed_at Claude's analysis output per review; primary key is (review_id, source)
ingestion_cursors source (PK), last_fetched_date Tracks the high-water mark per source to enable incremental ingestion on future runs
